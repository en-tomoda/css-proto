# CSAS AI機能 有識者レビュー用資料

**目的**: 本アプリ（CSAS: キャリアセレクタビリティシステム）のAI利用箇所について、有識者に以下を確認いただくための整理資料。

1. 実現可能性の確認
2. 利用技術の妥当性
3. 実装工数の確認
4. ランニングコストの超概算見積もり

現状は**全AI応答が固定文のモック**であり、本資料の工数・コストは社内のたたき台（ストローマン）です。数値の妥当性を含めてレビューをお願いします。

- アプリ全体仕様: [`career-support-app-spec.md`](./career-support-app-spec.md)
- モック: Next.js 16（このリポジトリ）。デモは `/login` からロールを選んで操作可能

---

## 0. 前提（想定規模の仮置き）

コスト・負荷の試算のため、以下を仮置きしています。**この前提自体もレビュー対象**です。

| 項目 | 仮置き |
|---|---|
| 利用メンバー数 | 100名（1社あたり） |
| 上司 | 10名 |
| 週次の利用イメージ | メンバー1人あたり: チャット2〜3セッション/週、振り返り1回/週 |
| 週次バッチ | アクション更新判定・提案: 全メンバー分を週1回 |
| SLA感 | 社内ツール水準（チャットは体感即応=ストリーミング必須、バッチは数分許容） |

## 2. 各機能の詳細

### A1. 初回設定: 仮目標・週次アクション生成

- **モック**: `app/onboarding/page.tsx`。5問（自己紹介/価値観/挑戦したいこと/避けたいこと/キャリアパス選択）→ 固定の仮目標＋アクション3件を表示
- **本実装イメージ**: 5問の回答＋選択キャリアパス＋（あれば）TA結果＋個社の昇進要件（A8）をコンテキストに、LLMが「仮目標1件＋今週のアクション3件」をJSONで生成。ユーザーが確認して確定
- **入出力の目安**: 入力 2〜4Kトークン / 出力 0.3〜0.5Kトークン
- **技術要素**: 構造化出力（JSONスキーマ強制）、プロンプトテンプレート
- **論点**:
  - 目標の品質（SMARTさ）と、キャリアパス・昇進要件との整合をどう担保するか
  - 生成のブレ対策（温度設定、few-shot例の管理）
  - 不適切回答（個人攻撃的な内容等）の入力時ハンドリング

### A2. AIチャット（目標相談・壁打ち）

- **モック**: `app/chat/page.tsx`。テンプレート「目標についてAIと話す」。応答は固定文ローテーション
- **本実装イメージ**: システムプロンプトに「本人の目標・今週のアクション・TAプロフィール・キャリアパス・個社昇進要件（A8）」を注入したマルチターン対話。ストリーミング応答
- **入出力の目安**: 1ターンあたり入力 2〜3K（コンテキスト込み）／出力 0.2〜0.4K。1セッション 8〜12ターン想定
- **技術要素**: チャットAPI＋ストリーミング（SSE）、会話履歴の永続化、プロンプトキャッシュ（固定コンテキスト部）
- **論点**:
  - コーチングとしての対話品質（アドバイスを急がず問いで返す等）のプロンプト設計と評価方法
  - 会話履歴が長期化した際の要約・圧縮戦略
  - キャリア相談というセンシティブ領域のガードレール（メンタル不調の兆候検知時のエスカレーション方針など）

### A3. AIチャット（週次振り返り）

- **モック**: A2と同一UI。テンプレート「振り返りを行う」で、今週のアクション達成状況を踏まえた固定の初手メッセージ
- **本実装イメージ**: A2と同基盤。今週のアクションと達成状況を注入し、「完了したこと→うまくいった要因→課題→次の一歩」の流れを誘導する対話。終了時にA4へ連結
- **入出力**: A2と同等
- **論点**: 振り返りの「終了」判定（明示ボタン想定）と、途中離脱時の扱い

### A4. 振り返りの要約

- **モック**: `components/reflection-list.tsx` ＋ `lib/data.ts` の `chatSummary`。「話したこと / 気づいたこと / 次の一歩」＋ひとことメモを固定データで表示
- **本実装イメージ**: 振り返りセッションの全会話ログを入力し、上記4項目のJSONを生成して保存。履歴画面（`/history`）とマイページで閲覧
- **入出力の目安**: 入力 3〜6K（会話ログ）／出力 0.3〜0.5K
- **技術要素**: 構造化出力。非同期実行（セッション終了時にバックグラウンドで生成）
- **論点**: 要約の忠実性（言っていないことを書かない）の担保・評価

### A5. 週次アクションの提案・差し替え

- **モック**: `lib/store.tsx`。3件完了で翌週新規提示（文言のみ）、キャンセル時は固定プールから代替を補充
- **本実装イメージ**:
  - **週次バッチ**: 各メンバーの目標・直近の振り返り・達成/キャンセル履歴（理由含む）を入力に、翌週のアクション3件を生成
  - **キャンセル時の差し替え**: キャンセル理由（選択＋自由記述）を反映した代替アクション1件を即時生成
- **入出力の目安**: 入力 2〜4K／出力 0.2〜0.3K
- **技術要素**: 構造化出力、ジョブスケジューラ（週次バッチ）、キャンセル理由のフィードバックループ
- **論点**:
  - 「物理的に不可能」「やりたくない」等の理由を次回生成にどう反映するか（単純なプロンプト注入で十分か）
  - アクションの粒度・実行可能性の品質評価（人手評価の設計）

### A6. 停滞アクションの深掘り誘導

- **モック**: `carriedWeeks >= 4` のルール判定でマイページに通知 → チャットへ誘導
- **本実装イメージ**: 検知はルールベースのまま（LLM不要）。チャット側で「このアクションが4週停滞している」文脈を注入したA2/A3の対話に接続
- **論点**: 追加のAIコストはほぼなし。通知文面の生成をLLM化するかは任意

### A7. 上司向けAIサマリー・接し方のヒント

- **モック**: `app/members/[id]/page.tsx` の `buildAiSummary()`。開示状態に応じてテンプレ文を組み立て
- **本実装イメージ**: **本人が開示を承認した情報のみ**＋TA結果＋アプリ活用状況を入力に、上司向けサマリーと接し方のヒントを生成。閲覧のたびに生成するとコスト・レイテンシがかさむため、**週次または開示状態変化時に生成してキャッシュ**する想定
- **入出力の目安**: 入力 2〜4K／出力 0.3〜0.5K
- **技術要素**: 構造化出力、キャッシュ戦略
- **論点（重要）**:
  - **情報境界の技術的担保**: 「未開示情報がプロンプトに一切入らない」ことをアプリ層で保証する設計（LLMに任せない）。ここの実装・テスト方針は有識者の見解が欲しいポイント
  - 人事評価に響きうる出力のトーン制御（断定を避ける等）と監査ログ

### A8. AIカスタマイズ（エン標準＋個社の昇進要件）

- **モック**: `app/admin/page.tsx`。「エン標準（CSAベース）」バッジ＋昇進要件のフリーテキスト設定（保存のみ）
- **本実装イメージ**:
  - エン標準（CSAのキャリア理論・コーチング方針）を共通システムプロンプト or ナレッジとして全AI呼び出しに適用
  - 個社の昇進要件テキストをテナント別にプロンプト注入
  - 要件が長文・複数文書になる場合はRAG（ベクトル検索）を検討。**当面はプロンプト注入で足りる見込み**（数百〜数千トークン想定）
- **論点**:
  - 「学習させる」というUI文言に対し、実態はプロンプト注入/RAGで足りるか、ファインチューニングが必要なケースはあるか（有識者確認）
  - マルチテナント時のプロンプト・設定の分離管理

## 3. 共通アーキテクチャの検討事項（有識者に確認したい）

1. **LLM API選定**: マネージドAPI（Anthropic Claude / OpenAI / Azure OpenAI / AWS Bedrock 等）前提。国内個社導入を見据えた場合のリージョン・データ取り扱い（学習不使用の契約条件）の推奨
2. **モデルの使い分け**: 対話（A2/A3）は中位モデル、要約・定型生成（A1/A4/A5/A7）は軽量モデル、という2階層で品質が足りるか
3. **構造化出力**: JSONスキーマ強制（tool use / structured output）で安定運用できるか、リトライ設計
4. **プロンプトキャッシュ**: 固定部（エン標準・個社要件・TAプロフィール）のキャッシュでコスト・レイテンシをどの程度圧縮できるか
5. **会話履歴の管理**: 永続化（DB）＋長期化時の要約圧縮の定石
6. **安全性**: キャリア/メンタル領域のガードレール、PII取り扱い、未開示情報の境界担保（A7）、監査ログ
7. **品質評価**: コーチング対話・生成アクションの評価手法（LLM-as-judge＋人手サンプリング等）の現実的な運用
8. **モック→本実装の差分**: 現在フロントのみ（インメモリ）。API層（Next.js Route Handlers or 別BFF）、DB、認証、ジョブ基盤の新設が前提

## 5. ランニングコスト超概算（要レビュー）

### 前提

- ユーザー100名、上司10名（1社）
- モデル価格は**2026年時点の一般的な水準の仮置き**（最新の公式価格で要再計算）
  - 軽量モデル級: 入力 $1 / 出力 $5（100万トークンあたり）
  - 中位モデル級: 入力 $3 / 出力 $15（同上）
- プロンプトキャッシュ・バッチAPI割引は**未考慮**（適用でさらに下がる余地）

### 月間トークン試算（1ユーザーあたり）

| 機能 | 回数/月 | 入力/回 | 出力/回 | 月間入力 | 月間出力 |
|---|---|---|---|---|---|
| チャット（A2/A3, 3セッション/週×10ターン） | 130ターン | 2.5K | 0.3K | 325K | 39K |
| 振り返り要約（A4） | 4.3回 | 4K | 0.4K | 17K | 2K |
| 週次アクション生成（A5） | 4.3回 | 3K | 0.3K | 13K | 1.3K |
| 上司サマリー（A7, 週次キャッシュ・按分） | 4.3回 | 3K | 0.4K | 13K | 2K |
| **合計/ユーザー/月** | | | | **約370K** | **約45K** |

### 100ユーザーでの月額（USD）

| 構成 | 月間トークン | 月額の目安 |
|---|---|---|
| すべて中位モデル | 入力37M / 出力4.5M | 37×$3 + 4.5×$15 ≈ **$180/月（約2.7万円）** |
| 対話=中位、生成/要約=軽量 | 同上（比率按分） | ≈ **$120〜150/月（約2万円前後）** |
| すべて軽量モデル | 同上 | 37×$1 + 4.5×$5 ≈ **$60/月（約1万円）** |

**ざっくり結論（たたき台）**: 100名規模で **月1〜3万円程度**（LLM API費のみ）。プロンプトキャッシュ活用でさらに2〜5割の圧縮余地。ユーザー数にほぼ線形にスケール。

※ 含まれないもの: インフラ（ホスティング・DB）、ベクトルDB（RAG採用時）、監視・ログ基盤、モデル価格改定。

### 有識者に確認したい点

- 利用頻度の前提（3セッション/週/人）は多すぎ/少なすぎないか
- モデル階層の使い分けと価格前提の妥当性
- プロンプトキャッシュ・バッチAPIでの現実的な圧縮率
